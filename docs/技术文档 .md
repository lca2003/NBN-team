# 技术文档

# 1\.整体架构

项目采用 Android 单 App、客户端单 `:app` 模块、Java 包分层的方案；仓库同时包含独立 Java 21 `:backend` 后端模块，用于承载 `/v1` 数据、互动、Stitch 页面 payload 和 AI 相关接口。训练营周期较短，Android 端没有拆成多个 feature Gradle 模块，而是通过包边界维护职责；后续如果项目长期维护，再按 `feature-feed`、`core-data`、`core-ai`、`core-analytics` 等方向拆客户端模块。

整体分层如下：

```Plain Text
UI 层
  Feed / Detail / Search / Stats
  负责页面展示、用户交互、列表滚动、卡片渲染和图表渲染

业务协调层
  FeedDataController / FeedInteractionDelegate / FeedExposureDelegate
  SearchBottomSheetDialogFragment / StatsDataLoader
  负责把 UI 事件转为 Repository、AI、Analytics、Video 模块调用

数据层
  AdRepository / DefaultAdRepository / MockAdRepository / RemoteDataSource
  负责广告分页、详情读取、搜索、互动状态和远程失败 fallback

AI 层
  AiSummaryService / AiTaggingService / AiSearchService / AiCache
  负责摘要、标签、聊天搜索、输出格式约束、缓存和降级

统计层
  AnalyticsTracker / ExposureTracker / AdAnalyticsEventStore
  负责曝光、点击、互动事件口径和 SQLite 明细记录

视频层
  VideoPlaybackManager / Media3VideoPlayerController
  负责播放状态、单播放器复用、暂停续播、静音和释放

后端服务
  8080 通用后端: Feed、详情、互动、Stitch、AI 搜索、AI 摘要/标签等 /v1 接口
  8081 AI 搜索运行面: SearchBottomSheet 默认本地 AI 搜索地址，走 /v1/ai/search，失败后本地 Repository 降级
  /api/ai/search: 早期 Spring AI 控制器路径，保留为实验/兼容口径，不作为当前默认后端主路径
```

核心原则是：UI 不直接读取 Mock JSON、不直接拼接网络请求、不直接持有 AI 模型细节；数据、AI、统计和视频能力通过稳定接口向 UI 提供服务。

## 1\.1关键设计原则

1. **单一数据入口**：Feed、详情、搜索结果和统计页统一通过 `AdRepository` 获取广告数据。互动状态也通过 Repository 更新，避免 Feed 点赞后详情页不同步，或搜索结果展示旧状态。

2. **AI 不阻断主流程**：摘要、标签和搜索都必须有缓存、规则或 Mock 降级。AI 失败时 App 仍能展示广告、进入详情和完成统计，不把模型服务变成主流程单点故障。

3. **事件先明细、后聚合**：曝光、点击、点赞、收藏、分享等事件先写入本地明细表，再由统计页聚合展示。这样比只维护计数器更容易扩展时间范围、来源页面、内容类型和标签热度。

4. **重资源集中管理**：视频播放只允许同一时间存在一个 active video。播放器由 `VideoPlaybackManager` 和 `Media3VideoPlayerController` 统一管理，避免 RecyclerView 中多个视频卡同时占用解码资源。

5. **端上可演示、远程可替换**：Mock 数据、raw 视频和本地规则保证离线演示稳定；Remote 数据源、8080 `/v1` 后端和 8081 AI 搜索运行面保留替换空间，方便联调真实服务。

## 1\.2主流程

首页主流程：

```Plain Text
MainActivity
  -> FeedFragment 初始化 RecyclerView、频道、搜索入口、统计入口
  -> FeedDataController 调 AdRepository.loadAds(...)
  -> Repository 返回分页广告和互动状态
  -> FeedAdapter 按内容类型渲染大图、小图、视频卡
  -> FeedExposureDelegate 监听可见区域并记录曝光
  -> FeedInteractionDelegate 处理点击、点赞、收藏、分享、视频播放
```

详情主流程：

```Plain Text
Feed 卡片点击
  -> 记录点击事件
  -> AdDetailActivity 通过 adId 读取同一 Repository 数据
  -> 展示图文、视频、统计、AI 摘要和 AI 标签
  -> 互动更新仍写回 Repository
  -> 返回 Feed 后列表位置和互动状态保持
```

AI 搜索主流程：

```Plain Text
SearchBottomSheetDialogFragment
  -> AiSearchService.search(query)
  -> RemoteAiSearchService 请求本地 AI 搜索地址
  -> POST /v1/ai/search
  -> 后端 envelope 返回 messages + results + provider/fallback
  -> Android DTO 适配为 answer + matchedAdIds + fallback
  -> Android 端按 matchedAdIds 从 Repository 回填广告卡片
  -> 远程不可用时使用本地 Repository 关键词搜索降级
```

统计展示主流程：

```Plain Text
曝光/点击/互动事件
  -> AnalyticsTracker
  -> AdAnalyticsEventStore SQLite 明细表
  -> StatsDataLoader 后台读取广告全集和事件汇总
  -> StatsSummaryBuilder 聚合指标
  -> StatsFragment 渲染指标卡、Top 广告、内容类型饼图、标签热度
```

# 3\.详细设计

## 人员A—信息流 UI 与交互——李诚昂

### 单列信息流与多样式广告卡片

#### 功能目标

信息流页面是整个广告推荐 App 的核心入口，负责向用户展示广告列表，并承载后续的详情页跳转、点赞收藏分享、标签筛选、AI 搜索结果展示、曝光点击统计和视频播放等功能。

该模块的目标不是简单展示一个列表，而是实现接近真实广告信息流产品的单列流式浏览体验。核心目标包括：

- 支持 RecyclerView 单列纵向滚动。

- 支持大图卡、小图卡、视频卡三种广告样式。

- 支持广告角标、品牌、标题、摘要、标签、统计信息和互动栏展示。

- 支持 cell 复用，避免长列表滚动时频繁创建 View。

- 支持后续 AI 标签、统计数据和互动状态接入。

- 保证信息流页面只负责 UI 展示和用户交互，不直接破坏成员B的数据层和成员C的统计、视频模块边界。

#### 难点与设计思考过程

##### 2\.1 广告卡片样式不同，但列表需要统一复用

信息流中有大图广告、小图广告和视频广告。三种卡片在布局上差异较大：大图卡更突出封面，小图卡偏信息密度，视频卡需要播放按钮、遮罩和视频角标。如果为每种卡片写完全独立的 Adapter 逻辑，会导致代码重复，也不利于后续维护。

因此采用 RecyclerView 多 viewType 方案。根据广告内容类型返回不同的 viewType，再绑定对应布局。三种卡片的公共字段尽量保持统一，例如标题、品牌、摘要、标签、统计信息和互动栏，这样可以复用部分绑定逻辑。

##### 2\.2 标签 chip 需要统一渲染，但不能强依赖 Material 组件

广告卡片和详情页都需要展示标签。如果直接引入 Material Chip，可能需要修改全局主题和 styles\.xml，而这些资源会影响其他成员模块，协作风险较高。

因此最终采用 TextView 自绘 chip 的方式，通过圆角 drawable、padding、文字颜色和背景色实现标签样式。这样不需要引入额外主题依赖，也方便在信息流和详情页复用。

##### 2\.3 UI 层需要先跑通，但不能直接改数据层

开发时成员B的数据层尚未提供完整分页接口，Mock 数据量也有限。如果直接修改成员B的 Repository 来支持分页，会破坏协作边界；如果完全等待数据层完成，成员A的信息流 UI 无法推进。

因此采用 UI 层分页垫片方案，在 `ui.feed` 内部新增 `AdCatalog`，通过现有 `AdRepository` 读取种子广告，再在 UI 层模拟分页、加载延迟和失败状态。后续成员B提供真实分页接口后，只需要替换 `AdCatalog.loadPage()` 的内部实现。

#### 方案对比

#### 最终方案与原因

最终采用 RecyclerView \+ LinearLayoutManager \+ 多 viewType 的方案实现单列广告信息流。`FeedAdapter` 根据广告类型区分大图卡、小图卡和视频卡，分别加载不同 XML 布局；公共字段保持一致，减少重复绑定逻辑。

标签渲染由 `TagChipBinder` 统一处理，避免每张卡片重复写标签 View 创建代码。后续重构后，广告卡片 ViewHolder 提取为 `FeedAdViewHolder`，底部加载区域提取为 `FeedFooterViewHolder`，使 `FeedAdapter` 只负责数据管理和 item 分发。

该方案的主要原因是：符合广告信息流多样式展示需求，同时保留 RecyclerView 的复用能力；代码边界清晰，后续可以继续接入真实图片、视频、标签筛选、搜索结果过滤和统计数据展示。

### 频道切换、刷新加载与列表状态管理

#### 功能目标

频道和刷新加载模块用于保证用户可以在不同广告频道之间切换，并能够通过下拉刷新、上拉加载获得连续的信息流浏览体验。

核心目标包括：

支持“精选 / 电商 / 本地”三个顶部频道 Tab。

切换频道后刷新当前频道广告列表。

再次点击当前 Tab 时列表回到顶部。

支持下拉刷新，刷新后回到列表顶部。

支持上拉加载更多，滚动接近底部时自动请求下一页。

支持加载中、空数据、加载失败、没有更多等状态展示。

支持频道、标签筛选、AI 搜索结果过滤之间的状态清理，避免多个筛选条件互相干扰。

#### 难点与设计思考过程

##### 2\.1 分页接口不完整，但 UI 必须展示完整加载流程

训练营项目要求展示下拉刷新和上拉加载，但早期 Mock 数据数量较少，也没有真实分页接口。如果没有分页能力，信息流只能展示少量广告，无法体现真实广告流体验。

因此在 UI 层增加 `AdCatalog` 作为分页适配层。它不直接取代成员B的数据层，而是调用已有 `AdRepository` 获取种子数据，再模拟分页、加载延迟、加载失败和没有更多状态。

##### 2\.2 频道、标签、搜索结果容易形成状态冲突

信息流后续需要接收标签筛选和 AI 搜索返回的广告 ID。如果频道切换、标签筛选、搜索结果同时生效，很容易出现空列表或用户无法理解当前列表为什么被过滤。

因此设计时将频道作为基础条件，标签和搜索结果作为附加过滤条件。切换频道或点击标签时，会清除旧搜索结果，避免搜索结果和标签条件叠加后造成状态混乱。

##### 2\.3 状态页需要和列表滚动解耦

加载中、空态、错误态如果直接混在列表 item 中，会导致状态控制复杂。尤其是空数据和加载失败时，列表可能没有 item，但页面仍需要展示提示和重试入口。

因此页面中单独设计状态层，底部加载更多使用 footer，整体状态和底部状态分开处理：页面级状态负责空态、错误态；列表 footer 负责加载中、没有更多和加载失败重试。

#### 方案对比

#### 最终方案与原因

最终采用 `FeedDataController + AdCatalog + FeedAdapter footer` 的组合方案。`FeedDataController` 负责频道切换、分页请求和状态切换；`AdCatalog` 负责从 Repository 获取广告并进行分页适配；`FeedAdapter` 负责展示广告 item 和底部加载状态。

该方案能够在不修改成员B数据层的情况下，完整实现频道切换、下拉刷新、上拉加载、空态和错误态。后续如果有真实分页接口，只需要替换 `AdCatalog` 内部数据来源，Feed 页面和 Adapter 基本不用改动。

### 广告详情页与跨页面互动状态同步

#### 功能目标

广告详情页用于承接首页广告卡片点击行为，展示更完整的广告内容，并支持点赞、收藏、分享等互动操作。详情页需要和首页保持状态一致，避免用户在详情页点赞后返回首页却看不到变化。

核心目标包括：

- 支持点击信息流广告进入详情页。

- 详情页展示封面、标题、品牌、摘要、标签、统计信息和互动栏。

- 支持图文广告和视频广告详情展示。

- 返回信息流后保持原列表滚动位置。

- 点赞、收藏状态在首页和详情页之间同步。

- 分享调用系统分享面板。

- 页面进入和返回具备基础切换动画。

#### 难点与设计思考过程

##### 2\.1 首页和详情页状态容易不一致

如果首页卡片和详情页各自维护点赞、收藏状态，就会出现详情页点赞后，首页仍然是未点赞状态的问题。广告信息流项目中，互动状态必须跨页面同步，否则会影响用户体验和统计准确性。

因此新增 `InteractionStore` 作为进程内单一状态源。首页卡片和详情页都通过广告 ID 读写同一份互动状态，避免每个页面维护自己的状态副本。

##### 2\.2 返回首页后不能丢失滚动位置

如果点击广告进入详情页后，返回首页时重新创建 FeedFragment 或重新加载列表，用户会被带回列表顶部，这不符合信息流浏览习惯。

因此详情页使用独立 `AdDetailActivity`，首页 `FeedFragment` 保留在返回栈中。返回时只刷新可见 item 的互动状态，不重建整个列表，从而保持原滚动位置。

##### 2\.3 详情页标签和首页标签需要一致

后期接入 AI 标签时，详情页如果异步加载 AI 标签并覆盖首页传入标签，就会出现同一条广告在首页和详情页标签不一致的问题。用户会觉得数据不统一。

因此修复时移除了详情页 AI 标签异步覆盖逻辑，详情页标签统一使用 Intent 传入的原始标签。AI 摘要可以异步加载，但标签展示保持和信息流一致。

#### 方案对比

#### 最终方案与原因

最终采用 `AdDetailActivity + InteractionStore + 局部刷新` 的方案。首页点击广告时跳转到详情页，详情页通过 Intent 接收广告基础信息并展示；点赞、收藏操作统一写入 `InteractionStore`。返回首页后，Feed 页面只刷新当前可见卡片状态，不重建列表。

该方案的主要原因是简单稳定，适合训练营阶段快速完成演示闭环；同时也保留了后续演进空间。如果后续接入正式后端，可以将 `InteractionStore` 替换或扩展为 Repository 层持久状态。

### 首页视频卡播放与播放器资源复用

#### 功能目标

视频广告是广告信息流中重要的卡片形态。早期视频卡只展示封面、遮罩和播放按钮，后续需要接入真实播放能力，并保证滚动列表中不会同时存在多个视频播放实例。

核心目标包括：

- 视频卡默认展示封面、遮罩、播放按钮和视频角标。

- 点击播放后显示 PlayerView 并播放视频。

- 再次点击或离开页面时暂停播放。

- 同一时间只允许一个 active video。

- 页面暂停或销毁时释放播放器资源。

- 避免 RecyclerView 复用导致视频播放状态错乱。

#### 难点与设计思考过程

##### 2\.1 RecyclerView 中不能每个视频卡都创建播放器

如果每个视频卡都创建独立 ExoPlayer，长列表滚动时会占用大量解码和内存资源，容易造成卡顿甚至崩溃。真实信息流中也通常会复用播放器资源。

因此视频播放接入成员C的 `VideoPlaybackManager` 和 `Media3VideoPlayerController`，由统一管理器控制当前 active video，保证同一时间只有一个视频处于播放状态。

##### 2\.2 播放结束后再次点击可能直接结束

开发过程中出现过视频播放一次后无法第二次播放的问题。原因是视频结束时保存了末尾进度，再次播放时 seek 到末尾，导致立刻触发结束。

修复方式是在 `onEnded` 和 `onError` 回调中将播放进度重置为 0，保证用户再次点击时可以从头播放。

##### 2\.3 详情页打开不能因为播放器初始化而卡顿

如果详情页一打开就创建 ExoPlayer，可能造成 100ms 到 300ms 的主线程阻塞，影响页面打开速度。尤其训练营演示时，页面卡顿会非常明显。

因此详情页视频采用懒初始化策略，只有用户点击播放时才创建播放器。AI 摘要请求也改为线程池处理，避免页面打开时创建大量新线程。

#### 方案对比

#### 最终方案与原因

最终采用 `PlayerView + Media3VideoPlayerController + VideoPlaybackManager` 的方案。视频卡默认显示封面和遮罩，点击后隐藏封面层并显示 PlayerView；同一时间只保留一个活跃视频。页面 `onPause` 时暂停播放，`onDestroyView` 时释放播放器资源。

该方案兼顾了真实播放能力和资源控制，符合信息流中“重资源集中管理”的原则，也避免多个视频卡同时播放造成解码资源浪费。现有技术文档整体架构中也将视频层定义为由 `VideoPlaybackManager / Media3VideoPlayerController` 负责播放状态、单播放器复用、暂停续播、静音和释放。

### 点赞收藏分享交互与动画设计

#### 功能目标

互动模块用于提升广告卡片和详情页的产品完成度，使用户可以对广告进行点赞、收藏和分享操作。除了状态变化，还需要提供一定的动效反馈，增强用户点击后的感知。

核心目标包括：

- 点赞支持灰色到红色状态切换。

- 收藏支持灰色到金色状态切换。

- 分享调用系统分享面板。

- 首页和详情页互动状态同步。

- 点赞支持动画反馈。

- 支持连续点赞时的连赞视觉效果。

- 收藏保持轻量反馈，避免动画过多造成干扰。

#### 难点与设计思考过程

##### 2\.1 动效不能只追求复杂，也要适合项目稳定性

开发中尝试过弹簧动画和 Lottie 动画。弹簧动画实现简单，但视觉效果不明显；Lottie 动画表现更丰富，但需要新增依赖和 JSON 资源，增加项目体积和维护成本。

最终选择用原生 `AnimatorSet + ObjectAnimator` 实现抖音式连赞特效，不依赖额外动画库，也能达到较明显的点击反馈。

##### 2\.2 连续点击时不能频繁取消点赞

如果用户快速连续点击点赞按钮，传统逻辑会在点赞和取消点赞之间来回切换，导致体验不符合短视频类产品中的“连赞”习惯。

因此设计为 1 秒窗口内重复点击不取消点赞，只继续播放爱心飞出动画；超过 1 秒后再次点击才取消点赞。这样既保留取消点赞能力，又提升连续点击时的娱乐性。

### 2\.3 动画不能影响业务状态

点赞动画只是视觉反馈，真正的点赞状态仍然由 `InteractionStore` 管理。如果动画和状态耦合过深，容易出现动画还在播放但状态已经变化的问题。

因此每次点赞动画都创建独立 ImageView，动画结束后从父布局移除。业务状态和动画生命周期分离，避免内存泄漏和状态错乱。

#### 方案对比

#### 最终方案与原因

最终采用原生动画实现点赞连赞特效。点赞时，爱心从按钮位置弹出，向上漂移、随机旋转、缩放并淡出；连续点击时多个动画实例互不干扰。收藏交互只保留颜色切换，分享调用系统分享面板。

该方案的主要原因是效果明显、实现可控、无额外依赖，适合训练营项目。它既能体现交互完成度，又不会引入过重的动画框架。

### 左右滑动切频道设计

#### 功能目标

除了顶部 Tab 点击切换频道外，信息流页面还增加了左右滑动切频道能力，让用户可以通过横向手势在“精选 / 电商 / 本地”频道之间切换，提升操作流畅度。

核心目标包括：

- 支持在 RecyclerView 卡片区域横向拖拽。

- 滑动过程中列表跟随手势移动。

- 滑动超过阈值后切换到相邻频道。

- 滑动未超过阈值时回弹到原位置。

- 首尾频道增加阻尼，避免越界切换。

- 解决 RecyclerView、SwipeRefreshLayout、视频 PlayerView 之间的手势冲突。

#### 难点与设计思考过程

##### 2\.1 横滑和纵向滚动容易冲突

信息流页面主要是纵向滚动，如果横滑识别过于敏感，会影响正常上下滑动；如果识别过于保守，又会导致用户很难触发频道切换。

因此最终通过 `RecyclerView.OnItemTouchListener` 判断手势方向。只有横向位移明显大于纵向位移时，才确认进入横滑频道切换逻辑。

##### 2\.2 子 View 可能抢占手势

视频卡中的 PlayerView、卡片中的按钮等子 View 可能会请求父布局不要拦截事件，导致横滑手势中断。

因此在 `FeedSwipeChannelDelegate` 中统一管理横滑手势。确认横滑后锁定当前手势链，并处理 `onRequestDisallowInterceptTouchEvent`，减少子 View 抢事件造成的断触。

##### 2\.3 切换动画需要有反馈

如果横滑后直接切换频道，用户会感觉页面突然变化。为了增强交互感，滑动过程中列表会跟手平移，松手后根据滑动距离执行飞出切换或回弹动画。

#### 方案对比

#### 最终方案与原因

最终新增 `FeedSwipeChannelDelegate` 管理左右滑动切频道。它负责手势识别、横滑锁定、跟手平移、边界阻尼、松手阈值判断、切换动画和回弹动画。FeedFragment 只负责创建并协调该委托，不直接写大量手势代码。

该方案的主要原因是体验更接近真实信息流产品，同时保持模块边界清晰。频道横滑逻辑被封装为独立类后，FeedFragment 代码也进一步瘦身，后续维护成本更低。

### Feed 模块重构与列表性能优化

#### 功能目标

随着信息流功能不断增加，FeedFragment 中逐渐包含频道切换、分页加载、曝光检测、互动处理、标签筛选、搜索过滤、视频播放和动画逻辑。如果继续堆在一个 Fragment 中，会导致文件过长、职责混乱、后续难以维护。

核心目标包括：

- 降低 FeedFragment 代码复杂度。

- 将频道、分页、曝光、互动、筛选、横滑等逻辑拆成独立模块。

- 减少 RecyclerView 滚动过程中的主线程耗时。

- 减少卡片重复绑定、图片重复加载和标签 View 重复创建。

- 避免 SQLite 读写阻塞 UI。

- 保证重构后功能不退化，并通过编译验证。

#### 难点与设计思考过程

##### 2\.1 Fragment 过大导致维护困难

FeedFragment 曾经达到 700 行以上，页面初始化、分页、互动、曝光、视频、筛选逻辑都集中在一个文件中。继续增加功能会导致排查 bug 困难，也不利于解释架构。

因此采用委托模式重构，将 FeedFragment 变成薄协调层，只负责视图初始化和各委托类连接。具体业务交给不同 Delegate 或 Controller。

##### 2\.2 滚动过程不能做耗时操作

RecyclerView 滚动时，如果频繁计算可见区域、写 SQLite、全量刷新列表、重复创建标签 View 或加载大图，都会造成明显掉帧。

因此做了多项性能优化：曝光写入移到后台线程；可见区域计算从 `getGlobalVisibleRect` 改为 `getLocalVisibleRect`；Glide 增加 override 尺寸限制；TagChipBinder 复用已有 View；统计数据读取异步化；用局部刷新替代 `notifyDataSetChanged`；滚动曝光检测增加 200ms 节流。

##### 2\.3 重构不能破坏队友模块

Feed 模块需要调用成员B的数据层、成员C的统计和视频能力，但不能直接修改他们的核心实现。否则合并时容易产生冲突，也不利于团队协作。

因此重构后所有新增类仍放在 `ui.feed` 范围内，通过公开接口调用 `AdRepository`、`AnalyticsTracker`、`VideoPlaybackManager` 和搜索入口，保持协作边界。

#### 方案对比

#### 最终方案与原因

最终采用“薄 Fragment \+ 多委托类”的结构。`FeedFragment` 只负责视图初始化和委托协调；`FeedDataController` 负责频道切换、分页加载和状态切换；`FeedExposureDelegate` 负责曝光检测；`FeedInteractionDelegate` 负责点击、点赞、收藏、分享、视频和动画；`FeedFilterManager` 负责标签筛选、搜索过滤和筛选栏 UI；`FeedSwipeChannelDelegate` 负责横滑切频道。

性能优化方面，将 SQLite 写入和统计读取移出主线程，减少滚动时 I/O；使用局部刷新减少无意义 rebind；复用 tag chip View；限制图片解码尺寸；滚动曝光检测节流。你的日志中也记录了 FeedFragment 从大文件拆成多个职责清晰类，以及滚动性能优化、视频接入、侧边滚动条和鼠标滚轮支持等改动。

该方案的主要原因是：在不大幅改动项目模块结构的前提下，明显提升了 Feed 模块的可维护性、滚动性能和可解释性，同时保持成员A、成员B、成员C之间的代码边界清晰。

### 数据模型与 Repository 统一入口

#### 功能目标

数据层是 Feed、详情页、搜索结果、统计页和 AI 摘要标签共同依赖的基础。如果每个页面各自读取 Mock JSON 或各自维护互动状态，就会出现首页、详情页、搜索结果数据不一致，也会让后续接入真实后端时需要大面积改 UI。

本模块的目标是把广告数据、分页、详情读取、搜索和互动更新统一收敛到 `AdRepository`，让 UI 只关心“拿到什么数据、当前是什么状态”，不关心数据来自本地 JSON、远程后端还是 fallback。

核心目标包括：

- 定义统一广告模型 `AdItem`，覆盖标题、品牌、频道、描述、摘要、图片、缩略图、视频、标签、统计和互动状态。

- 定义 `AdStats`、`InteractionState`、`InteractionAction`，统一曝光、点击、点赞、收藏、分享等字段和动作。

- 定义 `PageRequest`、`PageResult`、`SearchRequest`，支撑频道分页、刷新、加载更多和搜索分页。

- 定义 `DataResult`，统一表达成功、空态、超时、解析错误、远程错误和 fallback。

- 冻结 `AdRepository` 四个主入口：`loadAds()`、`getAdById()`、`searchAds()`、`updateInteraction()`。

#### 难点与设计思考过程

##### 2\.1 UI 不能直接读取 Mock JSON

如果 Feed 直接读 `ads_mock.json`，详情页再用 Intent 传完整广告对象，搜索页再自己复制一份过滤逻辑，短期看能跑，后续会立刻出现三类问题：字段变更需要改多个页面；点赞收藏状态无法同步；远程接口接入时 UI 要整体重写。

因此数据层采用 Repository 作为唯一入口。Feed 加载列表、详情页读取广告、搜索页获取结果、统计页读取广告全集，都通过同一个 `AdRepository` 完成。这样 UI 只依赖稳定接口，不关心底层数据源。

##### 2\.2 广告模型必须一次覆盖 Feed、详情、AI 和统计

早期广告模型只需要标题、品牌和摘要，但后续 Feed 需要图片和视频，详情页需要描述和 CTA，统计需要曝光点击等计数，AI 缓存又需要 `contentHash`。如果模型边做边补，很容易造成 A/C 联调时字段反复变动。

因此 `AdItem` 按“Feed \+ Detail \+ AI \+ Stats”全集字段设计。基础字段用于卡片展示，媒体字段用于图片和视频，统计字段用于曝光点击聚合，`contentHash` 用于 AI 缓存失效判断。字段多一些，但能减少后续接口返工。

##### 2\.3 错误态和 fallback 不能只靠异常

训练营演示环境可能出现后端未启动、真机无法访问本机端口、JSON 字段异常、远程响应为空等情况。如果数据层直接抛异常，UI 层就需要捕获大量细节，也容易导致 App 崩溃。

因此统一使用 `DataResult` 返回状态。`SUCCESS` 表示正常成功，`EMPTY` 表示没有数据，`TIMEOUT`、`PARSE_ERROR`、`REMOTE_ERROR` 表示不同失败类型，`FALLBACK` 表示远程失败但已经用本地数据兜底。UI 可以按状态展示加载态、空态、错误态或继续展示 fallback 数据。

#### 方案对比

方案 A：UI 直接读 Mock JSON。

- 优点是实现最快。

- 缺点是页面和数据源强耦合，状态同步和远程替换成本很高。

方案 B：每个页面各自维护数据适配层。

- 优点是页面推进灵活。

- 缺点是搜索、详情、统计会出现重复模型和重复状态。

方案 C：统一 Repository \+ 统一模型 \+ 统一结果状态。

- 优点是边界清晰，可测试，可替换，A/C 只需要依赖稳定接口。

- 缺点是前期需要先补模型、分页和错误语义。

#### 最终方案与原因

最终选择方案 C。人员 B 先冻结 Repository v1 和数据模型，再让 Feed、详情、搜索、统计、AI 都围绕这套接口接入。这样虽然前期需要多做模型和测试，但后续 A/C 不需要理解 Mock JSON 结构，也不需要关心 Remote 和 fallback 细节。

最终数据入口为 `AdRepository`。Feed 调 `loadAds(PageRequest)` 加载频道广告；详情页调 `getAdById(adId)` 读取同一广告；搜索页调 `searchAds(SearchRequest)` 获取匹配结果；曝光、点击、点赞、收藏、分享统一调 `updateInteraction(adId, action)` 更新状态和统计。

该方案的主要原因是稳定和可替换。当前可以使用本地 JSON 保证离线演示，后续接入后端时只替换 `RemoteAdDataSource` 或 `DefaultAdRepository` 装配方式，UI 层不用重写。

### Mock JSON、分页搜索与数据质量

#### 功能目标

Mock 数据不是简单占位，而是训练营演示和自动化测试的稳定事实源。它需要支撑首页频道、分页加载、详情页、搜索、标签、统计和 AI 降级，不能只有少量硬编码样例。

核心目标包括：

- 使用 `app/src/main/assets/ads_mock.json` 作为默认本地广告数据源。

- 覆盖“精选 / 电商 / 本地”三个频道。

- 覆盖大图、小图、视频三类广告内容。

- 覆盖运动、学生党、性价比、数码、通勤、本地、咖啡、学习、家居等常用标签和关键词。

- 覆盖长标题、空描述、低互动、高互动、视频字段等边界样本。

- 支持分页、频道过滤、关键词搜索、标签搜索和不存在 ID 的空态。

#### 难点与设计思考过程

##### 2\.1 硬编码数据不利于协作和测试

如果广告数据写死在 Java 代码里，A 想改一条卡片文案、C 想补搜索关键词、B 想增加边界样本，都需要改代码并重新编译。数据和逻辑混在一起，也不利于解释数据质量。

因此改为 JSON 数据集。`MockAdJsonParser` 负责解析 JSON，`MockAdRepository` 负责把解析后的广告放入内存 `adsById`。JSON 是事实源，Repository 是读写入口。

##### 2\.2 分页需要有稳定 cursor 和越界行为

信息流必须支持刷新和加载更多。如果分页只是返回完整列表，再由 UI 自己截取，会导致不同页面对分页口径理解不一致。

因此 `PageRequest` 使用 `page_1`、`page_2` 这类 cursor，`PageResult` 返回 `currentCursor`、`nextCursor`、`hasMore`、`pageNumber`、`pageSize` 和 `totalCount`。越界页返回 `DataResult.EMPTY`，并保留空 `PageResult`，方便 UI 显示“没有更多”或空态。

##### 2\.3 搜索要服务 C，但不能变成 AI 搜索替代品

本地 Repository 搜索的职责是稳定 fallback 和普通关键词过滤，不负责生成自然语言回答。它需要覆盖标题、品牌、频道、描述、摘要和标签，保证 AI 搜索服务不可用时仍能返回广告候选。

因此 `searchAds(SearchRequest)` 只返回广告分页结果，不生成聊天回复。对话式回复、推荐理由和 `matchedAdIds` 仍属于人员 C 的 AI 搜索链路。

需要注意的是，`BackendRemoteAdDataSource.searchAds()` 当前也不是专门的全量后端搜索接口，而是先请求后端 feed 候选集，再在 Android 侧做关键词或标签过滤。因此它适合作为普通搜索和 AI fallback 的稳定兜底，不等同于 `/v1/ai/search` 的对话式 AI 搜索能力。

#### 方案对比

方案 A：继续使用 Java Fixture。

- 优点是实现简单，不依赖 JSON 解析。

- 缺点是数据扩展和协作编辑困难。

方案 B：直接依赖远程后端数据。

- 优点是更接近生产。

- 缺点是演示强依赖后端启动和网络连通，离线不可控。

方案 C：本地 JSON 作为稳定事实源，Remote 作为可替换增强。

- 优点是离线可演示、测试可复现、字段可扩展。

- 缺点是需要维护 JSON 质量矩阵和解析测试。

#### 最终方案与原因

最终选择方案 C。当前本地广告数据集包含 30 条广告，覆盖三个频道和大图、小图、视频三类内容。频道数量不强行写死为完全平均，而是保证每个频道至少有足够样本支撑首屏、搜索和边界测试。

`MockAdRepository` 负责分页、搜索、详情和互动更新。它内部使用 `Map<String, AdItem>` 保存当前广告状态，既能保证读取效率，也能让 Feed、详情、搜索读到同一份状态。

该方案的主要原因是演示稳定。即使后端未启动，App 仍能打开首页、进入详情、搜索广告、展示统计和 AI fallback；同时 JSON 数据结构和后端 `/v1` 契约保持接近，后续替换远程数据成本较低。

### Remote 数据源、后端 HTTP 接入与 fallback

#### 功能目标

人员 B 的数据层不能只停留在本地 Mock。为了体现真实工程能力，需要保留后端 HTTP 接入边界，并在远程不可用时自动 fallback 到本地数据，保证演示主流程不被环境问题阻断。

核心目标包括：

- 定义 `RemoteAdDataSource`，抽象远程分页、详情、搜索和互动更新。

- 使用 `BackendRemoteAdDataSource` 对接 8080 后端 `/v1/feed`、`/v1/ads/{adId}` 和互动接口。

- 使用 `HttpApiClient` 封装 GET、POST、DELETE、PATCH 请求、超时、请求 ID 和响应读取。

- 使用 `BackendConfig` 配置候选地址，兼容模拟器 `10.0.2.2:8080` 和真机 `adb reverse` 后的 `127.0.0.1:8080`。

- 远程成功时返回后端数据，远程失败、响应异常或无可用数据时回落到本地 Mock。

#### 难点与设计思考过程

##### 2\.1 Android 真机和模拟器访问本机地址不同

模拟器访问电脑本机通常使用 `10.0.2.2`，真机则需要 `adb reverse` 后访问 `127.0.0.1`。如果只写死一个地址，演示环境一换就可能访问失败。

因此 `BackendConfig.defaultCandidates()` 提供多个候选地址。`BackendRemoteAdDataSource` 默认按候选地址尝试请求，全部失败后把错误交给 `DefaultAdRepository` 处理。

##### 2\.2 Remote 失败不能让首页白屏

后端可能未启动、端口被占用、状态文件异常、JSON envelope 缺字段，或者真机没有完成 `adb reverse`。这些问题都不应该让用户无法浏览广告。

因此 `DefaultAdRepository` 采用 Remote 优先、Mock fallback 的策略。远程返回成功且有数据时使用远程结果；远程抛出 `RemoteAdException`、返回空结果或不可用结果时，自动调用 `MockAdRepository`，并用 `DataResult.FALLBACK` 标记来源。

##### 2\.3 互动写操作需要优先同步后端，但不能丢失本地体验

点赞、收藏、点击、曝光、分享属于写操作。如果远程写入成功，应该读取后端返回的最新广告状态；如果远程写入失败，演示时仍要更新本地状态，避免用户点击后没有反馈。

因此 `DefaultAdRepository.updateInteraction()` 先尝试 `remoteDataSource.updateInteraction()`；远程失败时再调用 `MockAdRepository.updateInteraction()` 并返回 fallback。这样用户体验连续，也能说明“远程失败不会阻断主流程”。

#### 方案对比

方案 A：只保留本地 Mock。

- 优点是稳定。

- 缺点是无法体现真实 HTTP 接入和前后端分离边界。

方案 B：只使用后端 Remote。

- 优点是更接近真实服务。

- 缺点是训练营演示风险高，后端或网络异常会直接影响 App。

方案 C：Remote 优先 \+ Mock fallback。

- 优点是兼顾真实接口和演示稳定性。

- 缺点是需要在文档和 UI 状态中说明数据来源，避免把 fallback 误认为远程成功。

#### 最终方案与原因

最终选择方案 C。`RepositoryProvider.getRepository(Context)` 默认装配 `BackendRemoteAdDataSource.defaultDataSource()` 和 `MockAdRepository.fromAsset(context)`。正常情况下优先请求 8080 后端；后端不可用时自动回到本地 JSON。

后端侧已提供独立 `:backend` Java 21 服务，入口为 `BackendServer.main`，默认监听 8080，并通过 `/health` 和 `/v1/...` 提供数据。广告主链路包括 `/v1/feed`、`/v1/ads/{adId}`、点赞、收藏、分享、点击、曝光等接口。Android 侧通过 `BackendAdJsonParser` 把后端 JSON envelope 转为 `AdItem` 和 `PageResult`。

除广告主链路外，后端还提供 Stitch 页面 payload、用户、消息、评价评论、平台配置、素材和埋点等数据域。Android 侧对应保留 `BackendStitchDataSource`、`BackendUserDataSource`、`BackendMessageDataSource`、`BackendReviewDataSource` 和 `BackendPlatformDataSource` 等 client 入口。后端以 seed JSON 作为初始事实源，写操作进入 state 目录，既能保证训练营演示可复现，也能说明项目不是 UI 静态假数据。

该方案的主要原因是可演示、可联调、可解释。没有后端时 App 不崩；有后端时可以展示真正 HTTP 数据路径；后续迁移数据库或生产服务时，只需要继续保持 `/v1` 契约稳定。

### 跨页面互动状态同步

#### 功能目标

广告互动状态必须在 Feed、详情页、搜索结果和统计中保持一致。用户在首页点赞后进入详情页应该看到已点赞；在详情页收藏后回到首页和搜索结果也应该同步；曝光、点击、分享等统计也应该进入同一套状态更新口径。

核心目标包括：

- 点赞、取消点赞、收藏、取消收藏、点击、曝光、分享以及点赞/收藏 toggle 统一由 `InteractionAction` 表达。

- 所有状态更新都通过 `AdRepository.updateInteraction()` 触发。

- `MockAdRepository` 内部维护单一 `adsById` 状态源。

- 远程成功时以远程返回状态为准；远程失败时用本地状态保证演示反馈。

- Feed、详情、搜索结果读取同一广告 ID 时拿到一致的 `InteractionState` 和 `AdStats`。

#### 难点与设计思考过程

##### 2\.1 每个页面自己维护状态会造成不同步

如果 Feed 的点赞状态存在 Adapter 里，详情页的收藏状态存在 Activity 里，搜索页又持有另一份广告列表，用户切换页面后很容易看到互相矛盾的状态。

因此 B 的数据层把状态更新收敛到 Repository。UI 可以做即时动画和局部刷新，但最终状态以 Repository 返回的 `AdItem.interactionState` 和 `AdItem.stats` 为准。

##### 2\.2 互动动作既影响状态也影响统计

点赞和收藏会改变布尔状态，也会改变对应计数；点击、曝光和分享主要改变统计字段。如果这些逻辑散落在不同页面中，计数口径会不统一。

因此 `MockAdRepository.applyAction()` 统一处理 `LIKE`、`UNLIKE`、`TOGGLE_LIKE`、`COLLECT`、`UNCOLLECT`、`TOGGLE_COLLECT`、`CLICK`、`EXPOSE`、`SHARE` 等动作。UI 可以传 toggle 动作，Repository 或 Remote 数据源会根据当前状态解析为具体的点赞、取消点赞、收藏或取消收藏。状态和计数一起更新，避免 UI 自行永久加减。

##### 2\.3 搜索结果也必须继承最新互动状态

搜索结果不是独立广告副本。如果用户先在 Feed 点赞，再搜索同一广告，搜索卡片必须显示已点赞。否则用户会认为搜索页是旧数据。

因此搜索结果从 `adsById` 当前状态过滤出来，不重新构造脱离状态源的广告对象。单元测试也覆盖了 Feed、详情、搜索三处读取同一互动状态的场景。

#### 方案对比

方案 A：UI 自己维护互动状态。

- 优点是页面实现快。

- 缺点是跨页面同步困难，统计口径不可信。

方案 B：只用内存全局 Store，不经过 Repository。

- 优点是状态集中。

- 缺点是绕过数据层接口，后续 Remote 同步困难。

方案 C：Repository 作为状态写入口，内部维护本地或远程状态。

- 优点是本地、远程、测试路径统一。

- 缺点是 UI 每次状态变化都需要处理 Repository 返回结果。

#### 最终方案与原因

最终选择方案 C。所有互动动作统一经过 `updateInteraction()`，Repository 返回更新后的 `AdItem`。UI 可以根据返回结果刷新当前卡片、详情按钮和统计显示。

该方案的主要原因是状态可信。Feed、详情、搜索和统计都依赖同一数据源；Remote 成功时以后端为准，Remote 失败时以本地 fallback 为准。这样既满足训练营演示，也为后续真实用户关系、收藏列表和广告统计服务端化保留空间。

### AI 摘要和标签接口、缓存与降级

#### 功能目标

AI 摘要和 AI 标签用于详情页和广告卡片的信息增强，但不能成为主流程单点故障。即使后端 AI 不可用、模型未配置、返回为空或内容较弱，App 也要能展示可解释的摘要和标签。

核心目标包括：

- 定义 `AdAiService`，提供 `getAiSummary(adId)` 和 `getAiTags(adId)` 两个稳定入口。

- 使用 `AiResponse<T>` 返回值、来源、是否缓存、降级说明和错误对象。

- 支持 `remote_ai`、`cache`、`mock_fallback`、`rule_fallback` 四种来源。

- 摘要限制在 40 个中文字符以内。

- 标签限制为 3 到 5 个，每个标签不超过 6 个中文字符。

- 缓存 key 使用 `adId + contentHash + promptVersion`，内容变化后旧缓存自动失效。

#### 难点与设计思考过程

##### 2\.1 Android 端不应该持有模型密钥

如果 Android 直接调用云端模型，API Key、endpoint 和 model 都可能被反编译获取，不符合安全要求。

因此 Context 路径下的 AI 服务优先使用 `BackendAdAiService`，由后端 `/v1/ai/ads/{adId}/summary` 和 `/v1/ai/ads/{adId}/tags` 处理云端模型或规则 fallback。Android 端只消费结构化结果，不保存模型密钥。

当前 8080 `BackendServer` 的云端 AI 配置只从运行环境读取，使用 `NBN_AI_API_KEY`、`NBN_AI_ENDPOINT`、`NBN_AI_MODEL`。这些变量齐备时可以走可替换云端 provider；未配置或请求失败时返回规则降级。早期 Spring AI 控制器中的 `DASHSCOPE_API_KEY` 属于 8081 实验/兼容口径，不应写入 Android，也不作为当前默认后端主路径。

##### 2\.2 AI 输出必须有来源标记

如果只返回字符串，无法判断是模型成功还是降级结果。因此 `AiResponse` 中保留 `source` 和 `cached`。详情页可以展示 AI 来源，测试也可以验证第一次请求是 `remote_ai`，第二次请求变成 `cache`，弱内容广告走 `mock_fallback` 或 `rule_fallback`。

##### 2\.3 缓存不能误用旧内容

同一广告 ID 的文案、描述或标签可能变化。如果只按 `adId` 缓存，内容更新后仍可能展示旧摘要和旧标签。

因此缓存 key 加入 `contentHash` 和 `promptVersion`。摘要当前版本为 `summary_v1`，标签当前版本为 `tagging_v1`。广告内容或 prompt 版本变化后，旧缓存不会命中。

##### 2\.4 弱内容和异常输出必须兜底

部分广告可能描述为空，远程生成器可能抛异常，模型也可能返回空字符串或过长标签。AI 服务不能因为这些情况影响详情页打开。

因此摘要链路先查缓存，再尝试 remote generator，失败后使用 Mock 摘要，再不足时用标题、品牌、频道等规则生成。标签链路先查缓存，再尝试 remote generator，失败后使用广告已有标签，再不足时根据频道、内容类型和文案关键词生成规则标签。

#### 方案对比

方案 A：详情页直接调用模型或直接拼 prompt。

- 优点是看起来更“AI”。

- 缺点是密钥风险高、输出不可控、失败会影响页面。

方案 B：只使用 Mock 摘要和预置标签。

- 优点是稳定。

- 缺点是无法体现 AI 接口、缓存和降级工程能力。

方案 C：后端优先 \+ 本地 demo remote \+ 缓存 \+ Mock/rule fallback。

- 优点是安全、稳定、可测试，并能展示 `remote_ai -> cache -> fallback` 链路。

- 缺点是实现链路较长，需要清楚记录来源和约束。

#### 最终方案与原因

最终选择方案 C。`RepositoryProvider.getAdAiService(Context)` 默认返回 `BackendAdAiService(new DefaultAdAiService(repository))`。它先请求 8080 后端 AI 摘要和标签接口；后端不可用、响应异常或返回空内容时，再回到本地 `DefaultAdAiService`。

本地 `DefaultAdAiService` 通过 Repository 读取广告，默认注入 `DemoRemoteSummaryGenerator` 和 `DemoRemoteTagGenerator`。普通广告第一次请求返回 `remote_ai`，同一服务实例第二次请求命中 `cache`；弱内容或 remote 失败时继续走 Mock 或规则降级。

该方案的主要原因是“不阻断主流程”。AI 摘要和标签增强了详情页信息表达，但广告浏览、详情打开、搜索和统计都不依赖模型一定成功。输出来源、缓存命中和 fallback 原因都可以被测试和解释。

### 异常场景与降级策略

#### 功能目标

设计重点是把这些风险收敛在 Repository、Remote 和 AI service 内部，让 UI 不需要到处写异常处理。

#### 最终方案与原因

最终异常处理策略是：网络和后端问题由 Remote 层转换为 `RemoteAdException`，Repository 再转换为 `DataResult`；AI 问题由 `AiResponse` 标记来源和降级原因；UI 只根据状态展示结果。

### 给 A/C 的接口协作与验证

#### 功能目标

交付不是孤立的数据代码，而是给 A 和 C 提供稳定可依赖的接口包。A 负责页面和交互，C 负责搜索、标签过滤、统计展示，B 负责保证他们拿到的数据结构、状态和错误语义一致。

核心目标包括：

- 给 A 提供 Feed 卡片、详情页、媒体、互动状态和错误态所需字段。

- 给 C 提供搜索字段、标签字段、统计字段和可关联广告 ID。

- 给 UI 提供 `DataResult`，让加载态、空态、错误态和 fallback 态有明确依据。

- 给答辩提供接口契约、AI 设计、数据质量矩阵和阶段验证证据。

#### 难点与设计思考过程

##### 2\.1 接口变更会直接影响 A/C 进度

如果 B 后期频繁修改字段名、方法签名或搜索口径，A/C 会被迫返工。因此数据模型和 Repository v1 必须先稳定，再向 UI 暴露。

当前用 `docs/api-contract.md` 固定 Repository、分页、广告模型、互动、AI 摘要标签和后端 `/v1` 接口规则。UI 需要新增字段时，先更新契约和模型，再改页面。

##### 2\.2 测试要覆盖协作边界，而不是只测单个方法

数据层最容易出的问题不是某个 getter，而是跨页面状态、fallback、分页和缓存边界。因此测试重点放在 Repository、Mock 数据、Remote fallback、状态一致性和 AI 缓存降级。

当前测试覆盖了 JSON 可解析、分页首尾页和越界页、固定关键词搜索、不存在 ID 空态、Remote 失败 fallback、Feed/详情/搜索状态一致、AI remote/cache/mock/rule 来源和内容变化缓存失效。

##### 2\.3 文档要能支撑

只说“我写了数据层”不够。需要讲清楚为什么 UI 不直接读 JSON，为什么要 `DataResult`，为什么远程失败要 fallback，为什么 AI 缓存 key 不能只用 `adId`，以及真实后端和本地演示如何切换。

因此人员 B 还维护了 `docs/api-contract.md`、`docs/ai-data-design.md`、`docs/data-quality-matrix.md`、`docs/stage-completion-audit.md` 等材料，用于说明接口、数据覆盖、AI 降级和验证结果。

#### 方案对比

方案 A：只交代码，不写契约。

- 优点是短期省时间。

- 缺点是 A/C 联调容易靠口头约定，后期难定位责任边界。

方案 B：只写接口文档，代码后补。

- 优点是设计清楚。

- 缺点是缺少可运行验证，容易变成空文档。

方案 C：接口契约、实现代码、单元测试和验证日志一起交付。

- 优点是可联调、可回归。

- 缺点是需要持续维护文档和测试。

#### 最终方案与原因

最终选择方案 C。人员 B 的交付由四部分组成：数据模型与 Repository 实现、Mock/Remote/后端接入、AI 摘要标签缓存降级、测试与文档证据。

当前聚焦验证可通过数据层和 AI 相关单测，覆盖 Repository 与 AI 主路径。提交或最终交付前仍应按仓库要求使用 JDK 21 运行 `./gradlew assembleDebug`，保证 Android Debug 包可构建。

该方案的主要原因是协作成本低。A/C 不需要关心本地 JSON、后端地址、AI fallback 细节，只需要依赖 Repository、AI service 和契约文档；B 也能通过测试和日志证明自己的部分不是“能写出来”，而是“可复用、可替换、可验证”。

#### 验证证据索引

接口设计能力、异常降级能力、验证闭环能力。人员 B 的价值不只是提供数据，而是让 A/C 的页面和统计模块建立在稳定、可替换、可证明的数据底座上。

### **AI对话式搜索界面**

#### 功能目标

AI 聊天搜索用于在广告信息流中提供自然语言查询能力。用户可以在底部搜索浮层中输入类似“适合学生党的运动产品”“找咖啡相关广告”这类问题，系统返回一条 AI 回复，并把匹配到的广告 ID 交给 Feed 模块过滤展示。

该功能的目标不是让 AI 直接生成广告卡片，而是让 AI 负责“理解用户意图并返回匹配广告 ID”，广告数据、卡片渲染、互动状态和曝光统计仍由原有 Feed 与 Repository 体系负责。

核心目标包括：

- 支持聊天式搜索 UI 和自然语言输入。

- 支持远程 AI 服务匹配广告。

- 支持模型不可用、接口失败、地址不通时的本地搜索降级。

- 搜索结果与 Feed、详情页共享同一份广告数据和互动状态。

#### 难点与设计思考过程

##### Android 不能持有模型密钥

如果 Android 端直接调用大模型 API，模型 endpoint、model 和 key 都可能被反编译获取，不适合放进 APK。

因此最终把真实模型调用放到 8081 后端，Android 只访问项目自己的 `/api/ai/search`。8081 后端通过环境变量 `DASHSCOPE_API_KEY` 判断是否调用模型；未配置时直接返回 fallback。

##### 搜索结果要与信息流状态一致

广告卡片有点赞、收藏、分享、曝光、点击等状态。如果 AI 直接返回完整广告对象，就可能和 Repository 中的最新互动状态不一致，也会绕过 Feed 里的卡片动态化、统计和过滤逻辑。

所以搜索结果只返回 `matchedAdIds`。Android 端收到广告 ID 后，由 `MainActivity` 通知 `FeedFragment`，再由 Feed/Repository 体系按 ID 回填和过滤广告。

##### 远程服务和运行环境不稳定

训练营演示时，可能遇到后端未启动、模拟器和真机访问本机地址不同、模型 key 未配置、模型输出格式错误等问题。搜索入口不能因为这些问题直接不可用。

因此 Android 端设计了多候选地址和本地 Repository fallback：优先请求 `http://10.0.2.2:8081/`，失败后尝试 `http://127.0.0.1:8081/`；全部失败后走本地关键词搜索，并保持返回结构仍是 `answer + matchedAdIds + fallback`。

#### 方案对比

#### 最终方案与原因

最终选择方案 C。它在安全性、可演示性和架构边界之间比较均衡：真实 AI 能力由后端承载，Android 端只处理稳定的结构化响应；远程不可用时仍能通过本地 Repository 给出结果，不影响主流程演示。

AI 聊天搜索采用“前端对话浮层 \+ 8081 后端 AI 匹配 \+ Feed 按广告 ID 展示 \+ 本地搜索降级”的方案。Android 端通过 `AiSearchService` 抽象搜索能力，搜索 UI 只提交 query 并展示 AI 回复；远程实现 `RemoteAiSearchService` 依次请求多个 8081 候选地址，成功后接收 `answer`、`matchedAdIds` 和 `fallback`，失败后使用本地 `AdRepository` 做关键词搜索降级。

后端以 8081 Spring AI 服务 `/api/ai/search` 为主路径。后端从 seed 广告目录构造候选集，prompt 要求模型只返回 JSON，并在解析后用候选广告 ID 白名单过滤结果，避免模型幻觉广告 ID。Android 收到匹配 ID 后不直接渲染广告卡片，而是通知 Feed 模块按 ID 过滤广告，保证搜索结果与首页、详情页共享同一份 Repository 数据和互动状态。

该方案的主要原因是安全、稳定和边界清晰：模型密钥不进入 APK，AI 输出被后端和 DTO 双层约束，远程不可用时仍有本地降级；同时搜索模块只负责意图理解和结果 ID，广告卡片动态化、曝光点击统计和互动状态仍复用 Feed/Repository/Analytics 的既有能力。

### 标签点击过滤

#### **功能目标**

标签点击过滤用于提升信息流广告的结构化浏览能力。用户看到某个广告标签后，可以直接点击标签，在当前频道中筛选出同类广告；再次点击已选标签可以取消筛选，也可以通过顶部筛选栏移除单个标签或清除全部。

核心目标包括：

- 广告卡片上的标签可点击，并能触发当前 Feed 内部筛选。

- 支持多标签筛选，多标签之间采用 AND 语义。

- 当前筛选标签在顶部筛选栏可见，卡片内命中的标签高亮。

- 标签过滤与 AI 搜索结果过滤互斥，避免复杂叠加导致空列表难以解释。

- 过滤状态变化时复用 Feed 分页、卡片动态化和统计链路，不单独重做搜索结果页。

- 列表刷新尽量轻量，标签高亮变化不触发整卡重绑和图片重载。

#### 难点与设计思考过程

##### 标签点击应当是筛选，不是关键词搜索

标签是结构化字段，和标题、摘要、描述里的自然语言不同。如果点击“运动”时把标题或摘要里包含“运动”的广告也算命中，结果会更像关键词搜索，容易出现误筛。

因此当前口径选择只匹配 `AdItem.tags` 字段。`TagFilter.byTag()` 使用 `item.getTags().contains(tag)`，`TagFilter.byTags()` 使用 `item.getTags().containsAll(tags)`，不扫描标题、品牌、摘要或描述。

##### ViewHolder 不能直接操作数据源

标签 chip 的点击发生在 `FeedAdViewHolder` 内部，但真正的筛选状态属于页面级状态。如果让 ViewHolder 直接过滤列表或访问 Repository，会导致卡片绑定层承担过多职责，也不利于测试。

因此设计为事件上抛：`TagChipBinder` 只负责渲染和点击监听，`FeedAdViewHolder` 只把事件交给 `FeedInteractionListener.onTagClick()`，`FeedFragment` 再交给 `FeedFilterManager` 管理筛选状态。

#### 方案对比

#### 最终方案与原因

最终选择方案 C。在不新增后端接口的情况下完成标签过滤闭环，同时保留清晰的模块边界，后续如果数据量扩大，可以把 `TagFilter` 的规则迁移到 Repository 或后端。

标签点击过滤采用 Feed 内部筛选方案。广告卡片绑定时，`FeedAdViewHolder` 通过 `TagChipBinder` 把 `AdItem.tags` 渲染为可点击 chip；点击后通过 `FeedInteractionListener` 把标签文本上抛到 `FeedFragment`，再由 `FeedFilterManager` 管理当前筛选集合。筛选状态变化后，顶部筛选栏展示已选标签，卡片中的同名标签高亮，并触发 `FeedDataController` 刷新当前频道。

过滤数据流复用 Feed 分页链路。`FeedDataController` 调用 `AdCatalog.loadPageWithTags()` 时传入标签集合和搜索广告 ID；`AdCatalog` 从 Repository 获取分页广告后，先按搜索 ID 过滤，再按标签集合做 AND 过滤。真正的标签过滤由 `TagFilter` 完成，它只精确匹配 `AdItem.tags` 字段，不匹配标题、摘要或描述，从而保证标签点击是结构化筛选，而不是关键词搜索。

该方案的主要原因是边界清晰和演示稳定：ViewHolder 不直接操作数据源，FilterManager 只管理筛选状态，AdCatalog 只负责加载与过滤适配，TagFilter 保持纯函数规则。列表性能上，标签选中态通过 payload 局部刷新，只重绑 chip，不重载整张卡片；chip 子 View 复用和请求版本号也减少了滚动分配和陈旧结果覆盖风险。后续如果数据量扩大，可将同样的过滤口径下沉到 Repository 或后端接口。

### 曝光点击统计

#### 功能目标

曝光统计用于模拟广告信息流中的基础埋点能力：当广告卡片进入可视区域并达到有效曝光条件时，记录一次曝光事件；当用户点击卡片进入详情页时，记录一次点击事件；统计页再基于本地事件明细展示曝光、点击、转化率、互动率和 Top 广告等指标。

核心目标包括：

- 曝光不能简单等同于“卡片出现在列表中”，需要有可见比例和停留时长口径。

- 同一页面生命周期内，同一广告只记录一次有效曝光，避免上下滑动重复计数。

- 曝光、点击、点赞、收藏、分享等事件统一进入统计模块，方便后续聚合。

- 统计写入不能阻塞首页滚动和卡片点击主流程。

- 统计页能从本地事件明细中聚合可视化数据，而不是只依赖内存计数。

#### 难点与设计思考过程

##### 曝光口径需要避免误计

如果卡片只露出一小部分，或者用户快速滑过，直接计曝光会导致曝光量虚高，CTR 和后续分析也会失真。

当前实现选择“可见比例 \+ 连续停留时间”的口径：广告卡片可见面积比例必须大于 50%，并连续停留 1000ms，才算一次有效曝光。注意这里是严格大于 50%，等于 50% 不触发。

##### 滚动列表中不能频繁做重计算和 I/O

RecyclerView 滚动时，如果每一帧都计算可见比例、写 SQLite 或同步 Repository，容易造成掉帧。

因此当前方案把曝光检测做节流，并把 SQLite 写入放到后台线程。UI 状态先即时更新，统计持久化异步执行，保证滚动体验优先。

##### 统计数据需要跨页面和刷新保留

仅用内存计数会在页面重建、进入统计页或 App 生命周期变化时丢失，也不方便统计页聚合。

因此当前方案使用 SQLite 追加事件明细表。曝光、点击和互动事件都作为明细写入 `ad_analytics_events`，统计页通过 GROUP BY 汇总。

##### 事件记录不能影响业务操作

埋点失败不应该阻止用户浏览、点击详情或点赞收藏。如果 SQLite 写入失败、Repository 统计更新失败，业务 UI 仍应继续响应。

因此 `AnalyticsTracker` 的持久化会捕获异常，`FeedExposureDelegate` 和 `FeedInteractionDelegate` 也先更新本地 `InteractionStore` 和 UI，再异步写统计。

#### 方案对比

#### 最终方案与原因

最终选择方案 C，并配合本地 SQLite 明细表。这个方案在演示稳定性、统计可信度和实现复杂度之间比较平衡：端上能离线工作，曝光口径又比简单计数更可靠。

曝光统计采用“前端可视检测 \+ 纯状态机判定 \+ 本地事件明细 \+ 统计页聚合”的方案。首页通过 `FeedExposureDelegate` 监听 RecyclerView 滚动、停止、页面恢复和数据加载完成等时机，遍历当前可见卡片并计算可见面积比例；曝光口径由 `ExposureTracker` 统一判断，默认要求广告卡片可见比例大于 50%，连续停留 1000ms，同一 tracker 生命周期内同一广告只记录一次曝光。

达到曝光口径后，系统先更新 `InteractionStore` 中的曝光数并刷新当前卡片，再通过 `AnalyticsTracker` 异步写入 SQLite 明细表，同时调用 Repository 的 `updateInteraction(EXPOSE)` 模拟同步 Mock 或远端广告统计。点击事件定义为“从 Feed 卡片进入详情页”，由 `FeedInteractionDelegate` 在跳转详情前记录，点赞、收藏、分享作为独立互动事件，不混入点击。

该方案的主要原因是统计口径更可信，同时不影响列表性能。曝光判断拆成纯逻辑状态机，便于测试 50% 阈值、1000ms 停留和去重边界；滚动后计算最早统计时间，根据剩余时间延迟复查；SQLite 写入和统计汇总都放到后台线程。统计页再通过 `StatsDataLoader` 和 `StatsSummaryBuilder` 聚合总曝光、总点击、转化率、互动率、Top 广告和内容类型占比，保证本地模拟统计可展示、可追踪、可扩展。

### 统计数据展示

#### 功能目标

统计数据展示用于把 Feed 中产生的曝光、点击、点赞、收藏、分享等事件转成可读的统计面板。用户从底部导航进入统计页后，可以看到核心指标、热门广告排行、内容类型占比和标签热度。

核心目标包括：

- 展示总曝光、总点击、CTR、点赞、收藏、分享、互动率等核心指标。

- 展示热门广告 Top 5，并用条形占比表达曝光差异。

- 展示大图、小图、视频三类内容占比。

- 展示标签热度，帮助观察广告内容标签分布。

- 数据读取不阻塞 UI，统计聚合逻辑可测试。

- 简单图表满足演示需求，避免引入重量级第三方图表库。

#### 难点与设计思考过程

##### 统计数据来源不只一个

统计页需要同时读取广告元数据、Feed/详情共享的互动状态、SQLite 事件汇总。如果只看广告对象初始 stats，Feed 中新产生的曝光和点击可能无法反映到统计页；如果只看 SQLite，也缺少标题、标签、内容类型等展示元信息。

因此最终把统计数据定义为“广告全集 \+ 交互状态 \+ 事件汇总”的合成结果。`StatsDataLoader` 负责加载广告全集和事件汇总，`StatsSummaryBuilder` 再结合 `InteractionStore` 做聚合。

##### 数据读取不能阻塞页面进入

统计页需要读 Repository 和 SQLite，尤其 SQLite 汇总会执行 GROUP BY。如果直接在主线程读取，进入统计页时可能卡顿。

因此设计了 `StatsDataLoader`：后台线程读取广告全集和事件汇总，再通过主线程 Handler 回调给 `StatsFragment` 渲染。

##### 聚合逻辑要和 UI 解耦

统计指标有排序、百分比、Top 限制、缺失类型处理、重复计数处理等边界。如果这些逻辑直接写在 Fragment 中，不利于测试，也会让页面代码膨胀。

因此聚合逻辑集中在纯 Java 类 `StatsSummaryBuilder`，输出只读模型 `StatsSummary`。`StatsFragment` 只负责把聚合结果渲染成指标卡、条形占比、饼图和标签 chip。

##### 统计页必须有空态和边界处理

统计可能在没有曝光、没有点击、某类广告数量为 0 时进入页面。如果没有处理，会出现除 0、空白图表或列表撑不开的问题。

当前处理方式是：百分比除数为 0 时返回 0；Top 广告无曝光时显示空态文案；饼图无有效切片时绘制灰色圆形占位；内容类型缺失时返回 0。

#### 方案对比

#### 最终方案与原因

最终选择方案 B。它和当前项目的本地统计事件链路最匹配：端上可以离线展示统计结果，聚合逻辑可单测，UI 实现轻量。

统计数据展示采用“后台加载 \+ 纯 Java 聚合 \+ 原生轻量图表”的方案。统计页由 `StatsActivity` 承载 `StatsFragment`，从底部导航进入；`StatsDataLoader` 在后台线程读取广告全集和 SQLite 事件汇总，再回到主线程渲染，避免 Repository 和数据库读取阻塞 UI。

统计聚合集中在 `StatsSummaryBuilder`。它把广告列表、`InteractionStore` 和持久化事件汇总合成为只读 `StatsSummary`，包含总曝光、总点击、CTR、点赞、收藏、分享、互动率、内容类型占比、Top 广告和标签热度。Top 广告按曝光数倒序、点击数倒序、广告 ID 升序排序，最多展示 5 条；标签热度按出现次数倒序、名称升序排序，最多展示 6 个。

图表展示没有引入第三方库。内容类型饼图由 `PieChartView` 使用 `Canvas.drawArc()` 绘制，空数据时绘制灰色占位圆；热门广告曝光条使用 `LinearLayout` 权重表达相对占比，适配不同屏幕宽度；标签热度使用动态 chip，每三枚自动换行。该方案依赖少、实现轻、可测试性好。

### AI输出格式约束与缓存策略

#### 功能目标

AI 聊天界面用于承接用户的自然语言搜索输入，例如“找咖啡相关广告”“适合学生党的运动产品”。界面需要展示一条 AI 回复，并把后端匹配出的广告 ID 交给 Feed 模块过滤展示。

本模块的关键目标不是让 AI 直接生成广告卡片，而是把 AI 输出约束成稳定的搜索结果结构：

- `answer`：展示给用户看的自然语言回复。

- `matchedAdIds`：Feed 可识别的广告 ID 列表。

- `fallback`：标记本次结果是否来自降级逻辑。

缓存策略也只围绕聊天界面展开：当前不缓存搜索结果，只持久化聊天消息历史。这样既能恢复用户关闭浮层前的对话，又能避免旧搜索结果和广告候选集变化产生不一致。

#### 难点与设计思考过程

##### AI 输出天然不可控

大模型可能返回普通段落、非法 JSON、空 `answer`、重复广告 ID，甚至返回当前广告库中不存在的 ID。如果 Android 直接消费模型原文，聊天界面可能无法解析结果，Feed 也无法根据不存在的广告 ID 展示卡片。

因此设计时把模型输出压缩到固定 schema：`answer + matchedAdIds + fallback`。其中 `matchedAdIds` 还需要在 8081 后端经过候选广告白名单过滤，保证传给客户端的 ID 来自项目真实数据源。

##### 搜索结果必须回到 Feed 体系

广告卡片的动态样式、点赞收藏状态、曝光点击统计都在 Feed、Repository 和 Analytics 模块中维护。如果聊天界面直接接收 AI 生成的广告对象，就会绕开这些已有状态，导致搜索结果卡片和首页卡片不一致。

因此 AI 聊天界面只消费广告 ID：聊天浮层展示 `answer`，宿主 `MainActivity` 把 `matchedAdIds` 转交给 `FeedFragment`，最终仍由 Feed 按 ID 过滤和渲染卡片。

##### 搜索结果缓存容易产生旧数据

自然语言 query 的组合很多，同一句 query 在不同广告候选集下也可能匹配到不同广告。如果客户端按 query 缓存搜索结果，广告数据更新、推荐策略调整或后端 prompt 变化后，用户可能看到过期匹配。

因此当前不做搜索结果缓存。相比缓存搜索结果，本项目更需要保证搜索结果跟随当前广告目录和后端策略实时变化。聊天界面只持久化消息历史，用于恢复对话展示，不把它当成下一次搜索的结果缓存。

#### 方案对比

#### 最终方案与原因

最终选择方案 C。它把 AI 的职责限定为“理解 query 并返回推荐理由和广告 ID”，把展示、互动状态和统计仍留在 Feed 体系中；同时不缓存搜索结果，避免旧匹配污染当前广告列表。

AI 聊天界面的输出格式约束采用“8081 后端结构化响应 \+ Android DTO 兼容解析 \+ Feed 按 ID 展示”的方案。Android 搜索浮层只提交用户 query，不直接调用大模型；8081 `/api/ai/search` 负责模型调用、JSON 解析、`answer` 空值校验、`matchedAdIds` 白名单过滤和 fallback 标记，最终返回 `answer + matchedAdIds + fallback`。Android 收到结果后只展示 `answer`，并把 `matchedAdIds` 上抛给 Feed，由 Feed 继续按 Repository 数据渲染广告卡片。

缓存策略上，AI 聊天搜索当前不缓存搜索结果。原因是自然语言 query 组合多、广告候选集会变化、prompt 策略可能调整，按 query 缓存容易造成旧广告 ID 污染当前结果。项目只对聊天消息做本地 SQLite 持久化，保存消息文本、是否用户消息、匹配广告 ID 和创建时间，用于搜索浮层关闭后恢复历史；用户再次发送相同 query 时仍会重新走 8081 或本地 fallback 搜索链路。
