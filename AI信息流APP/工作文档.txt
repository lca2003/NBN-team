# 成员A工作流文档：广告信息流 UI 与交互实现

## 一、成员A职责定位

成员A主要负责广告推荐信息流 App 的**核心页面展示与用户交互体验**，重点完成用户能直接看到、点击、滑动、切换、刷新、进入详情、点赞收藏分享等功能。

在整个“AI广告推荐信息流”项目中，成员A承担的是**客户端核心体验层**，对应课题要求中的核心功能：

1. 单列广告信息流布局
2. 大图、小图、视频三类广告卡片
3. 顶部频道 Tab 切换
4. 下拉刷新、上拉加载更多
5. 点击广告进入详情页
6. 返回后保持列表位置
7. 点赞、收藏、分享交互 UI 与动效
8. 配合成员B的数据层和成员C的统计/搜索功能完成页面联动

---

## 二、成员A整体工作流

成员A的工作可以按照下面 6 个阶段推进。

---

## 阶段一：需求拆解与页面结构确认

### 目标

明确信息流页面需要实现哪些 UI、哪些交互、哪些状态，避免一开始就直接写代码导致结构混乱。

### 需要完成的内容

1. 阅读课题要求，提取与成员A相关的核心需求。
2. 确认页面包含：

   * 信息流首页
   * 广告卡片
   * 顶部频道 Tab
   * 广告详情页
   * 点赞/收藏/分享区域
   * 刷新和加载状态
3. 和成员B确认广告数据字段，例如：

   * `id`
   * `title`
   * `brand`
   * `channel`
   * `contentType`
   * `summary`
   * `imageUrl`
   * `videoUrl`
   * `tags`
   * `stats`
   * `interaction`
4. 和成员C确认需要触发哪些统计事件，例如：

   * 广告点击
   * 点赞
   * 收藏
   * 分享
   * 进入详情页

### 产出物

```text
docs/member-a-workflow.md
docs/ui-page-structure.md
```

建议你在文档里写清楚：

```text
成员A负责信息流 UI 与交互，包括首页 Feed、广告卡片、频道切换、刷新加载、详情页、互动按钮和基础动画。
UI 层不直接处理网络请求和 AI 调用，所有广告数据通过 Repository 获取，互动事件通过 analytics 模块上报。
```

## 阶段二：信息流首页 Feed 页面搭建

### 目标

完成广告信息流的主体页面，让用户打开 App 后能看到单列广告列表，并且列表可以正常滚动。

### 页面结构

建议首页结构如下：

```text
FeedActivity / FeedFragment
│
├── 顶部频道 Tab
│   ├── 精选
│   ├── 电商
│   └── 本地
│
├── 下拉刷新容器 SwipeRefreshLayout
│
└── RecyclerView 单列广告列表
    ├── 大图广告卡片
    ├── 小图广告卡片
    └── 视频广告卡片
```

### 推荐包结构

```text
ui/
  feed/
    FeedActivity.java
    FeedAdapter.java
    FeedViewHolder.java
    LargeImageAdViewHolder.java
    SmallImageAdViewHolder.java
    VideoAdViewHolder.java
    FeedUiState.java
```

### 具体任务

1. 创建 `FeedActivity` 或 `FeedFragment`。
2. 使用 `RecyclerView` 实现单列列表。
3. 使用 `LinearLayoutManager` 控制垂直滚动。
4. 创建 `FeedAdapter`。
5. 根据广告类型返回不同 ViewHolder。
6. 列表中展示广告标题、品牌、摘要、图片、标签、点赞收藏状态等内容。

### 对应课题要求

这一部分对应：

```text
实现单列广告信息流布局，支持流畅滚动与列表位置保持。
```

### 注意点

不要把所有卡片样式都写在一个 ViewHolder 里，否则代码会变得很乱。建议按照广告类型拆分：

```text
LARGE_IMAGE -> LargeImageAdViewHolder
SMALL_IMAGE -> SmallImageAdViewHolder
VIDEO       -> VideoAdViewHolder
```

这样更符合“卡片动态化方案”的要求，也方便答辩时说明设计理由。

## 阶段三：多样式广告卡片实现

### 目标

完成至少三种广告卡片样式，满足课题中“大图、小图、视频”广告卡片的要求。

---

## 1. 大图广告卡片

### 适用场景

适合品牌广告、生活方式广告、电商大促广告，视觉冲击力强。

### 页面元素

```text
大图广告卡片
├── 品牌名
├── 广告标题
├── 大图
├── AI 摘要 / 广告摘要
├── 标签列表
├── 点赞按钮
├── 收藏按钮
└── 分享按钮
```

### 实现重点

1. 图片宽度占满卡片。
2. 标题和摘要层次清晰。
3. 标签横向排列。
4. 底部互动区固定样式。

---

## 2. 小图广告卡片

### 适用场景

适合本地生活、电商商品、密集信息展示。

### 页面元素

```text
小图广告卡片
├── 左侧文字区
│   ├── 标题
│   ├── 品牌
│   ├── 摘要
│   └── 标签
│
└── 右侧小图
```

### 实现重点

1. 图片放右侧，文字放左侧。
2. 控制文字行数，避免撑高卡片。
3. 保证滚动列表的视觉整齐。

---

## 3. 视频广告卡片

### 适用场景

适合短视频广告、产品展示、活动推广。

### 页面元素

```text
视频广告卡片
├── 视频封面
├── 播放按钮
├── 静音图标
├── 标题
├── 摘要
├── 标签
└── 点赞 / 收藏 / 分享
```

### 实现重点

1. 外流视频默认暂停。
2. 点击播放按钮后播放。
3. 滑出屏幕后暂停或释放。
4. 进入详情页后可以支持自动播放或继续播放。
5. 视频资源不要每个卡片都单独创建，后续要配合 `video.player` 做播放器复用。

### 对应课题要求

```text
广告卡片支持多样式：大图、小图、视频。
外流视频默认暂停，点击按钮播放，进入详情页支持播放控制。
```

## 阶段四：频道 Tab、刷新和分页加载

### 目标

实现信息流常见的基础交互，让广告列表更接近真实广告推荐信息流产品。

---

## 1. 顶部 Tab 切换

### 页面结构

```text
顶部频道：
精选 | 电商 | 本地
```

### 交互逻辑

1. 默认进入“精选”频道。
2. 点击“电商”，清空当前列表。
3. 请求或读取电商频道广告数据。
4. 刷新 RecyclerView。
5. 点击“本地”，同理切换本地频道。
6. 切换频道时显示加载状态。

### 和成员B的协作接口

成员A调用：

```text
AdRepository.getAds(channel, cursor, pageSize)
```

成员B负责返回对应频道数据。

### 对应课题要求

```text
顶部 Tab 支持切换频道，如精选 / 电商 / 本地，切换时刷新数据。
```

---

## 2. 下拉刷新

### 交互逻辑

1. 用户下拉列表。
2. 显示刷新动画。
3. 重新请求第一页数据。
4. 替换当前列表。
5. 停止刷新动画。

### UI 状态

```text
正常状态
刷新中状态
刷新成功状态
刷新失败状态
```

### 实现建议

使用：

```text
SwipeRefreshLayout
```

---

## 3. 上拉加载更多

### 交互逻辑

1. 用户滑动到列表底部。
2. 触发加载更多。
3. 根据 `nextCursor` 请求下一页数据。
4. 将新数据追加到列表末尾。
5. 如果 `hasMore = false`，显示“没有更多了”。

### 页面状态

```text
加载中
加载成功
加载失败
没有更多
```

### 对应课题要求

```text
支持下拉刷新、上拉加载更多。
```

## 阶段五：广告详情页与返回位置保持

### 目标

实现从信息流点击广告进入详情页，返回后仍停留在原位置，满足真实信息流体验要求。

---

## 详情页页面结构

```text
DetailActivity
│
├── 广告标题
├── 品牌信息
├── 大图 / 视频
├── 摘要
├── 智能标签
├── 曝光数 / 点击数
├── 点赞按钮
├── 收藏按钮
└── 分享按钮
```

---

## 跳转逻辑

### Feed 页面点击卡片

```text
点击广告卡片
    ↓
记录当前 RecyclerView 滚动位置
    ↓
传递 adId 到 DetailActivity
    ↓
DetailActivity 根据 adId 获取广告详情
```

### 返回逻辑

```text
用户从详情页返回
    ↓
Feed 页面恢复之前的频道
    ↓
恢复 RecyclerView 滚动位置
    ↓
同步点赞 / 收藏状态
```

---

## 位置保持方案

建议保存：

```text
currentChannel
firstVisiblePosition
offset
```

可使用：

```text
LinearLayoutManager.findFirstVisibleItemPosition()
View.getTop()
```

返回后调用：

```text
layoutManager.scrollToPositionWithOffset(position, offset)
```

### 对应课题要求

```text
点击卡片进入详情页，返回保持原位置。
```

## 阶段六：点赞、收藏、分享交互与动画

### 目标

完成信息流和详情页的互动体验，同时保证状态同步。

---

## 1. 点赞功能

### 交互逻辑

```text
未点赞 → 点击 → 点赞成功 → 图标高亮 → 数量 +1
已点赞 → 点击 → 取消点赞 → 图标恢复 → 数量 -1
```

### 动画建议

1. 图标放大缩小。
2. 点赞按钮颜色变化。
3. 可以加一个轻量弹跳动画。

---

## 2. 收藏功能

### 交互逻辑

```text
未收藏 → 点击 → 收藏成功
已收藏 → 点击 → 取消收藏
```

### 注意点

收藏状态需要在 Feed 和 Detail 中同步。

---

## 3. 分享功能

### 交互逻辑

```text
点击分享按钮
    ↓
弹出分享提示 / 系统分享面板 / Toast
    ↓
记录分享事件
```

训练营项目中可以先做本地模拟，不一定真的接入完整分享 SDK。

---

## 4. 状态同步

### 推荐方案

互动状态统一放在数据层，由成员B的 Repository 管理，成员A只负责调用和刷新 UI。

```text
Feed 页面点赞
    ↓
Repository 更新 interaction.liked
    ↓
Feed 局部刷新
    ↓
进入 Detail 时读取同一份状态

Detail 页面收藏
    ↓
Repository 更新 interaction.collected
    ↓
返回 Feed 后列表显示同步后的状态
```

### 对应课题要求

```text
数据、状态的传递，比如详情页和信息流的点赞状态同步。
```

# 七、成员A答辩时可以这样讲

你可以直接按这个逻辑介绍自己的工作：

> 我主要负责广告信息流的 UI 与交互部分，包括单列信息流首页、三种广告卡片样式、顶部频道切换、刷新加载、详情页跳转、返回位置保持，以及点赞、收藏、分享等互动功能。
>
> 在实现上，我没有把所有 UI 逻辑堆在一个页面里，而是按照 `ui.feed` 和 `ui.detail` 进行拆分。Feed 页面使用 RecyclerView 实现单列滚动列表，并根据广告的 `contentType` 区分大图、小图和视频三种 ViewHolder，这样后续如果要增加新的广告样式，只需要新增一个 ViewHolder，不需要大改原有代码。
>
> 在交互体验上，我实现了频道 Tab 切换、下拉刷新和上拉加载更多。点击广告卡片后进入详情页，返回时通过保存 RecyclerView 的当前位置和偏移量恢复列表位置，保证用户不会丢失浏览进度。点赞和收藏状态不在页面中单独维护，而是通过统一的数据状态同步，保证 Feed 页面和详情页看到的是同一份互动状态。
>
> 这个设计比较符合广告信息流的核心要求：列表滚动要流畅、卡片样式要可扩展、交互状态要一致、详情页返回体验要自然。
